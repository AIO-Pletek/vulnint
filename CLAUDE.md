# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

VulnInt is a vulnerability intelligence & management platform for hosting/provider infrastructure. It ingests CVE data from upstream feeds (NVD, CISA KEV, Ubuntu USN, Debian DSA, AlmaLinux/Rocky/CloudLinux errata, MSRC, cPanel, ExploitDB), correlates findings against the installed-package inventory of managed Linux and Windows servers, and dispatches alerts.

## Commands

All commands run from the repo root and wrap `docker compose`:

```bash
make build            # Build all Docker images
make up               # Start stack (detached)
make down             # Stop stack
make restart          # Restart all services
make logs             # Tail logs (all services)
make ps               # Service status
make shell-api        # Open shell in the API container
make shell-db         # psql into Postgres
make migrate          # Run `alembic upgrade head` inside the API container
make revision m="..." # Create an autogenerated Alembic revision
make seed             # Seed initial admin user + default roles
make reindex          # Rebuild the OpenSearch CVE index from Postgres
make test             # Run backend tests (`pytest -q` inside the API container)
make lint             # Run ruff + mypy inside the API container
make clean            # Remove all volumes (destructive)
```

API interactive docs are available at `/docs` when the stack is running.

## Architecture

```
CVE feeds ──► Celery workers ──► Postgres ◄── FastAPI ◄── JWT ── Next.js frontend
                                  ▲
Linux / Windows agents ──► inventory ingest (X-Agent-Token)
                                  │
                                  └──► OpenSearch (CVE full-text search index)
                                       │
                                       ▼
                                  Notification channels (email, Telegram, Discord, Slack, SIEM)
```

### Backend (`backend/`)

- **Framework**: Python 3.12, FastAPI, SQLAlchemy 2 (async with asyncpg), Alembic
- **Entry point**: `backend/app/main.py` — creates the FastAPI app, registers middleware (CORS, rate limiting via slowapi, security headers, global exception handlers), and mounts the v1 API router at `/api/v1`
- **Config**: `backend/app/core/config.py` — all settings loaded from env via pydantic-settings, cached with `@lru_cache`
- **Database**: `backend/app/core/database.py` — async engine, `AsyncSessionLocal` sessionmaker, and a `get_db` FastAPI dependency. Models use `TimestampMixin` (created/updated/deleted_at) and `UUIDPrimaryKey`
- **Auth**: `backend/app/auth/deps.py` — `get_current_user` (JWT bearer → DB lookup with eager-loaded roles/permissions) and `require_permissions(*codes)` dependency factory. `backend/app/auth/permissions.py` — static catalog of permission codes (`server:read`, `cve:write`, etc.) and default seed roles (admin, analyst, viewer, agent)
- **API routes**: `backend/app/api/v1/` — one module per resource (auth, servers, inventory, cves, correlations, alerts, alert_rules, feeds, audit, dashboard, users, roles, health). The router in `backend/app/api/router.py` aggregates them all
- **Repositories**: `backend/app/repositories/` — data-access classes per aggregate (CVERepo, AdvisoryRepo for CVE upserts; ServerRepo, InventoryRepo, CorrelationRepo for server/inventory lifecycle). Uses raw SQLAlchemy statements, not a generic base — each repo owns its queries
- **Workers**: Celery with Redis broker. `backend/app/workers/celery_app.py` defines the app, routes tasks to queues (`feeds`, `correlation`, `alerts`, `default`), and configures the beat schedule. Feed connectors in `backend/app/workers/feeds/` follow a `BaseFeed` ABC with retry, throttling, and a `FeedRecord` normalized dataclass
- **Versioning**: `backend/app/utils/versioning.py` — dpkg and RPM version comparison, `is_vulnerable(installed, fixed, os_family)`, and `kb_matches` for Windows KB matching. These are the core of the correlation engine

### Frontend (`frontend/`)

- **Framework**: Next.js 14 App Router, TypeScript, Tailwind CSS, Radix UI primitives
- **Pages**: `/login`, `/dashboard/*` (overview, servers, CVEs, alerts, correlations, feeds, settings, users). Server detail and CVE detail have dynamic `[id]` routes
- **Auth**: JWT stored client-side, sent as Bearer to the API. The layout checks auth state

### Agents (`agents/`)

- **Linux**: Python 3 (stdlib only), installed via `install.sh` into `/opt/vulnint/`, runs via systemd timer every 6h. Enumerates packages with dpkg/rpm, POSTs to `/api/v1/inventory`
- **Windows**: PowerShell 5.1+, installed via `Install-Agent.ps1`, runs as a Scheduled Task. Enumerates KBs via `Get-HotFix` and installed software from registry (avoids `Win32_Product` to prevent MSI self-repair storms)
- Both agents use `X-Agent-Token` header (hashed bearer token, SHA-256 stored server-side, never returned after creation)

### Key flows

1. **Feed ingestion**: Beat schedules periodic tasks → feed connectors fetch upstream sources → normalized `FeedRecord` objects → `CVERepo.upsert()` into Postgres → OpenSearch reindex task syncs modified CVEs
2. **Agent inventory**: Agent POSTs package list → `InventoryRepo.create_with_packages()` stores inventory + packages → correlation task runs (`correlation.run_all`) matching installed packages against affected products → `CorrelationRepo.upsert()` creates/updates correlations
3. **Alerting**: `alerts.dispatch_pending` runs every 5 minutes, checks open correlations against alert rules, dispatches to configured notification channels

### Risk scoring (`backend/app/utils/scoring.py`)

`compute_risk_score()` combines CVSS base (0-80), KEV (+15), exploit availability (+8), production exposure (+7), and server count (log-scaled, capped at +10) into a 0-100 score. This is a heuristic — not a replacement for EPSS/SSVC.

### RBAC model

Permissions are granular action:resource codes. Users belong to roles; roles grant permissions. The `require_permissions()` dependency checks the union of all role permissions. Superusers bypass all checks. Seed roles:
- **admin**: all permissions
- **analyst**: read servers/CVEs/correlations/alerts/audit, write correlations
- **viewer**: read-only
- **agent**: inventory ingest only

### Schema conventions

- All primary keys are UUIDs
- Soft-delete via `deleted_at` timestamp on most models
- API schemas (Pydantic) are in `backend/app/schemas/`, separate from ORM models in `backend/app/models/`
- Async session is yielded per-request via `get_db` — always roll back on exception
